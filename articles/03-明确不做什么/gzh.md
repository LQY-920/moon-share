# 比"明确做什么"更难的是"明确不做什么"

> 本文是"moon-share"系列第 03 篇。是 [01 四层设计法](../01-四层设计法/gzh.md) 和 [02 关键决策表](../02-关键决策表/gzh.md) 的姐妹篇。

## 一、先讲一个场景

你让 AI 帮你设计一个账户系统。

AI 给了你一份 3000 字的设计文档：有表结构、有 API、有测试策略。看起来很专业。

你让 AI 开工。三天后一个陌生的 `verify_email` 表出现在数据库里——但你并没有要邮箱验证功能。

你问 AI：为什么加了这个？

AI 说："邮箱验证是账户系统的常见功能，我帮您预留了。"

你又气又无奈。这种"热情帮你多做一点"的 AI，每个开发者都领教过。

**根本原因**：你的设计文档里只写了"要做什么"，没写"**不做什么**"。

## 二、两份真实文档的反例证据

我翻了一下自己最近两份 spec。发现一件事：**它们都在显眼位置有一节专讲"不做"**。

这不是巧合。

### 证据 1 · 平台愿景文档（L1 Vision 层）

在第七节"战略性说明"的末尾，有一个子章节叫：

> **暂不纳入的讨论点（保留但不立项）**
> - 独立的开发者/扩展 SDK（给平台写插件）— 近期不独立，按需嵌入各子系统
> - 独立的通知/触发器子系统 — 归入 S4.6 工作流 和 S4.7 Agent
> - 独立的多端同步/离线子系统 — 归入 S2.1 存储 和 各 L4 形态

三项。每一项都写清楚：**不做什么 + 为什么不做 + 改由谁承担**。

### 证据 2 · Identity 子系统设计文档（L2 Design 层）

在第一节"定位与边界"的 1.2 小节，有一张表：

```
| 事项                        | 归属                         |
|-----------------------------|------------------------------|
| 角色/权限策略引擎            | S1.2 Policy                  |
| 审计事件的订阅与存储         | S1.3 Audit (S1.1 只发布事件)  |
| 计费与配额采集               | S1.4 Billing                 |
| OAuth/SSO 具体 provider 接入 | 延后子系统 (S1.1 只留接口层)  |
| 邮箱发送服务（验证邮件、找回密码） | 延后（M0 只留占位路由）       |
| 前端登录/设置页面            | 前端形态（L4）负责            |
```

六项。每一项都写清楚：**本模块不做 + 归谁做**。

然后在第六节"会话机制"的末尾，还有一个小节：

```
| 主动排除                  | 原因                                    |
|--------------------------|------------------------------------------|
| Refresh token            | opaque token + 滑动活跃时间已覆盖        |
| CSRF token               | 同域 + SameSite=Lax 足够；分域后再加      |
| Session 绑定 IP/UA 硬校验 | 切换网络易误杀；只记录不强校验            |
| 并发会话数上限            | M0 不设限制，合规要求出现再加             |
```

还有第九节"测试策略"的 9.2：

```
9.2 明确不做
- CI 内跑安全扫描器（ZAP/Burp） —— 单人项目过度
- Fuzzing
- Benchmark —— 登录 QPS 非 M0 瓶颈
```

**两份文档，合计 4 处"不做"小节，共 16 项明确排除。**

这不是写 spec 的副产品，这是 spec 的主产品。

## 三、为什么"不做"比"做"信息量更大？

这一节讲理论，讲清楚你就明白了为什么两份 spec 都花大篇幅写"不做"。

### 信息量的本质：排除可能性的数量

信息论里有个直觉：**一个陈述的信息量，等于它排除了多少其他可能性**。

"我要做一个网站" → 信息量很小，因为网站有千百种。
"我要做一个面向 C 端的订阅制内容平台" → 信息量大，因为排除了 B 端、排除了免费模式、排除了电商。

同样的道理：

"登录系统要做多设备会话" → 信息量一般。
"登录系统要做多设备会话，**但不做并发会话数上限、不做 IP 绑定、不做 CSRF token**" → 信息量大。

**第二句把边界画死了，AI 读完第二句比读完第一句知道的多得多**。

### "做"很容易，"不做"很难

当你写"要做 X"时，你只需要能想象出 X 的样子。
当你写"不做 Y"时，你需要先想到 Y 存在，然后判断它不值得做，然后给出理由。

**三重思考**。

这就是为什么大多数 AI 生成的设计文档只有"要做什么"——因为那是 AI 最擅长的（列清单）。AI 最不擅长的是"主动告诉你它为你排除了什么"，因为这要求它先假装自己是不同的人，分别主张不同方案，再替你做排除。

大多数人写文档也一样。写"不做"比写"做"累 3 倍。

### "不做"抵御三种常见故障

**故障 1 · AI 的"热情膨胀"**

AI 的训练目标之一就是"给出全面回答"。遇到"设计账户系统"的请求，它的倾向是把所有能想到的功能都加上：邮箱验证、密码强度提示、双因素认证、找回密码、账户合并……

没有"不做"清单，这些全会被默默加进来。

**故障 2 · 时间久了你自己的记忆模糊**

三周后你回来加新功能，AI 问你："要不要顺便加 refresh token？"

你说："当时是不是讨论过？……好像有……好像没有……"

如果文档里写着：
> 主动排除：Refresh token —— opaque token + 滑动活跃时间已覆盖

你 3 秒钟内就能回答"不加"。

**故障 3 · 跨子系统的"职责偷渡"**

AI 设计"账户系统"时，会倾向于把审计也一起做了（"反正数据库里加个 log 表很容易"）。把权限也做一半（"先简单判断一下 is_admin 吧"）。

一段时间后你想独立做审计子系统，发现账户表里已经埋了审计字段，两边各自写日志，数据重复、不一致。

Identity 文档的 1.2 小节那张"不做的事"表，就是防这个的。**"审计事件的订阅与存储归 S1.3 Audit，本模块只发布事件"**——一行字，从源头掐断职责偷渡。

## 四、怎么写"不做"小节

### 基本格式：两列或三列

**两列格式（最小版）**：

```
| 事项 | 归属/原因 |
```

**三列格式（推荐）**：

```
| 不做什么 | 为什么不做 | 未来什么条件变了可以重新考虑 |
```

第三列可选，但对于"当前不做、未来可能做"的项强烈建议加。

### 写"不做"的三个好位置

不是把所有"不做"全堆在文档末尾。有三个好位置：

**位置 1 · 紧跟"定位与边界"小节**

放在 spec 第一节。让读者（包括 AI）在读任何具体设计前，先知道**整个模块的边界**。

S1.1 Identity 文档的 1.2 小节就在这个位置。

**位置 2 · 具体技术决策的章节末尾**

比如在"会话机制"章节讲完了"我们用什么"之后，附一个"主动排除"小节，讲"我们没用什么"。

**好处**：排除项和正向决策在同一个上下文里，读者立刻明白为什么排除。

**位置 3 · 测试策略的 "明确不做"**

测试最容易陷入"什么都想覆盖"。列出"不做的测试类型"，让测试工作量收敛到合理范围。

### 写"不做"的三条质量标准

**标准 1 · 每项都要带理由**

没有理由的"不做"等于没说。

- ❌ `不做 JWT`
- ✅ `不做 JWT —— opaque token + 滑动活跃时间已经覆盖"长登录"需求`

**标准 2 · 理由要有具体参照，不是空话**

- ❌ `不做 Fuzzing —— 过度工程`（"过度工程"是结论不是理由）
- ✅ `不做 Fuzzing —— 单人项目，登录端点表面积小，人工审查成本低于 Fuzz`

**标准 3 · 区分"永久不做"和"当前不做"**

- 永久不做：写清楚为什么永不会需要（比如 DoS 风险、违反核心原则）
- 当前不做：写清楚"什么条件变了会重新考虑"（比如"多实例部署时"、"有企业客户时"）

## 五、反向用法：用"不做清单"倒推需求

这个技巧很多人没意识到。

当你对一个模糊的需求拿不准时，**反向列"不做清单"，比正向列"要做清单"容易得多**。

举个例子。老板说："给我们的内部工具加一个账户系统。"

你正向想："要做什么呢？登录吧？注册吧？重置密码吧？……"—— 越想越发散。

**反向想**：

- 不做 SSO（我们就一个应用）
- 不做团队概念（内部工具不分组）
- 不做公开注册（内部人手动添加）
- 不做审计完整版（有日志就够了）
- 不做密码找回（忘了找管理员重置）
- 不做邮箱验证（内部邮箱可信）

写完这 6 条"不做"，"要做什么"就浮出水面了：**一个仅邮箱密码登录、管理员后台加人、简单 session 的迷你账户系统**。

**"不做清单"是"需求定义"的反面最短路径**。

## 六、一个反模式：用"延后做"代替"不做"

这是个常见陷阱。

有些人（和有些 AI）会把所有"不做"都写成"延后做"：

```
| 延后做                    | 原因                |
|---------------------------|---------------------|
| 双因素认证                 | V2 考虑             |
| OAuth 登录                 | V2 考虑             |
| 找回密码                   | V2 考虑             |
| 组织/团队                  | V2 考虑             |
```

这个看起来很贴心——"不是不做，是以后做"。但它的问题是：

1. **它不减少需求，只是把需求推进冰柜**。心理上你还记挂着"这些 V2 要做"，决策时还会受它影响。
2. **"V2 考虑"等于没说**。什么条件触发 V2？没写。于是永远不触发。

正确的做法是区分两种情况：

**情况 A · 真的延后做**（已知会做，缺条件）：

```
| 延后做 | 触发条件 |
|---|---|
| OAuth 登录 | 有企业客户签约，要求 SSO 时 |
| 审计独立存储 | 审计事件量 > 10 万条/天 时 |
```

**情况 B · 主动排除**（判断永远不需要 或 现在的替代方案足够好）：

```
| 主动排除 | 为什么 |
|---|---|
| 双因素认证 | 内部工具，登录失败限流已足够 |
| 密码定期过期 | NIST 2024 明确反对此做法 |
```

"延后做"要有条件。没条件的"延后做"全部降级为"主动排除"，按主动排除的格式重写。

## 七、这个方法在非开发场景怎么用

任何需要定义范围的场景都适用。

**产品需求文档**：
```
| 本版本不做 | 原因 |
| 微信小程序适配 | 主 PMF 验证集中在 Web |
| 视频上传 | 带宽成本 + 审核复杂度，MVP 不验证此假设 |
```

**团队 OKR**：
```
| 本季度不做 | 原因 |
| 新市场拓展 | 存量市场渗透率 < 30%，优先加深 |
| 大版本重构 | 影响 Q2 交付节奏 |
```

**个人学习计划**：
```
| 本月不学 | 原因 |
| Rust | 现在学会跟不上本职工作节奏 |
| Go | 已经会了 Java，新语言优先级低于深耕 AI |
```

## 八、总结

- 比"明确做什么"更难的是"**明确不做什么**"
- 两份真实的 spec 文档，分别用了 3 处和 4 处"不做"小节，合计 16 项明确排除——**这不是副产品，是主产品**
- "不做"的信息量比"做"大得多（排除的可能性数量决定信息量）
- 写"不做"需要三重思考：想到它存在 → 判断不值得 → 给理由
- 抵御三种故障：AI 热情膨胀、时间后记忆模糊、跨子系统职责偷渡

**质量标准**：
1. 每项带理由
2. 理由要有具体参照
3. 区分"永久不做"和"当前不做（附触发条件）"

**反模式**：
不要把所有东西都写成"延后做"。没触发条件的延后做 = 主动排除，要按主动排除格式重写。

**反向技巧**：
需求模糊时，列"不做清单"比列"要做清单"容易得多。写完 6 条"不做"，真正的需求就浮出水面了。
