# 关键决策表：让 AI 头脑风暴不跑偏的那张三列表

> 本文是"moon-share"系列第 02 篇。上一篇 [四层设计法](../01-四层设计法/gzh.md) 提了一下这张表，这一篇专门讲它。

## 一、先看两张真表

### 表 1 · 来自 moon-agent-os 平台愿景文档（L1 Vision 层）

```
| 维度       | 决策                      | 说明              |
|-----------|--------------------------|-------------------|
| 目标用户    | 通用产品/平台(给别人用)    | 必须包含账户、数据隔离、分发等完整平台能力 |
| 日抛语义    | 按日进化                  | 软件瞬时、记忆持久 |
| 技术底座    | 构建在现有生态之上(编排层) | 模型/MCP/Claude Code 等都作为底层依赖 |
| 多形态关系  | 意图优先 + 形态优先 双路径 | 两种用户入口并存 |
| 记忆模型    | 用户级统一记忆中枢         | 单一归属主体，跨产物共享 |
| 分发模型    | 有公共市场(类 GPTs Store) | 创作者生态作为长期愿景的一部分 |
```

### 表 2 · 来自 S1.1 Identity 子系统设计文档（L2 Design 层）

```
| 维度          | 决策                        | 排除的替代              |
|---------------|----------------------------|-------------------------|
| 登录基线      | 仅邮箱+密码                 | 手机号验证码、OAuth、固定 .env 账号 |
| 会话机制      | Opaque token + sessions 表 | JWT、JWT + refresh      |
| Cookie 策略   | HttpOnly/Secure/SameSite=Lax | 跨域 SameSite=None（延后） |
| 密码哈希      | argon2id                   | bcrypt、scrypt          |
| 限流          | 两层 (IP + 邮箱),仅计失败   | 账户锁定策略（易被 DoS） |
| 注册入口      | CLI 创建账号                | 公网开放注册            |
| 实现路径      | 自写 + 成熟库               | Auth0/Supabase、Lucia   |
| 技术栈        | Node + TS + Express 5      | Python/FastAPI、Go、Java |
| 主键类型      | ULID                       | 自增 int、UUID v4       |
| ...（13 行）   | ...                        | ...                     |
```

这两张表不是随手列的，是两次 brainstorm 的核心收敛产物。Vision 层 6 行，Design 层 13 行，每一行背后都是 5-20 分钟的拉扯。

**这篇文章就讲一件事：为什么这张三列表是 AI 时代 brainstorm 最强的收敛工具**。

## 二、先讲反例：没有这张表会怎样

你和 AI brainstorm 一个登录系统。AI 很配合，问你要什么。你回答。它再问。你再答。

聊了一个小时，聊出一堆东西。你让它总结，它输出了一个 3000 字的文档。

第二天你回来看，发现问题：

**问题 1 · 每个决定只剩结论，没有"排除了什么"**

文档里写"登录用邮箱+密码"。

然后三天后你开始写代码，AI 问你"要不要也支持手机号登录？数据库表可以预留字段"。

你：啊，当时有讨论过吗？……好像讨论过……要不就加吧。

于是你加了 `phone` 字段、`phone_verified` 字段，写了短信发送逻辑。三周后你发现短信通道成本太高，要删掉。已经写了 200 行代码。

**根本原因**：当时 brainstorm 的结论只记录了"用邮箱"，没记录"明确排除了手机号"。文档没有"排除列"，AI 就不知道哪些是已排除的。

**问题 2 · 决定是"技术细节"，不是"维度"**

文档里写："用 JWT 做会话。密码用 bcrypt。限流用 express-rate-limit。"

看起来很具体。但三周后当你想把登录系统搬到别的项目，发现：

- 别的项目不用 Node.js，JWT 还能用吗？（其实你该问的是"为什么是 JWT 不是别的"）
- 别的项目限流需求不一样，express-rate-limit 还合适吗？（你该问的是"我的限流维度要求是什么"）

**根本原因**：没有"维度列"。AI 直接给你决策细节，你失去了"这个决定在回答什么问题"的线索。

## 三、三列表的结构：维度 / 决策 / 排除的替代

这张表的威力来自它的**形状**：

```
| 维度 (Dimension)  | 决策 (Decision)  | 排除的替代 (Alternatives Ruled Out) |
```

三列都必须有。缺一列威力就塌一半。

### 维度列：这个决定在回答什么问题

维度是**抽象的问题**，不是具体的技术。

- ❌ 错的维度写法：`JWT 还是 Session`
- ✅ 对的维度写法：`会话机制`

为什么？因为维度是**不随技术栈变**的。"会话机制"这个问题在 Node、Python、Go 里都存在，只是答案不同。如果你换技术栈，维度列不变，只需要重评决策列。

维度列的作用：

- **让你看到这个问题存在**。很多决定之所以翻车，是你压根没意识到这是个问题，AI 替你做了默认选择。
- **让维度之间可以相互检查**。比如"会话机制"选了 Opaque token，"Cookie 策略"就必须配套考虑。两个维度写在一起，矛盾一眼看出来。

### 决策列：**一行一个结论**，不含"也"、"可能"、"考虑"

决策列必须是斩钉截铁的陈述。

- ❌ 错的写法：`使用 JWT，也考虑 session`
- ✅ 对的写法：`Opaque token + 服务端 sessions 表 + Cookie`

"考虑"、"也许"、"可能"这些词出现在决策列里，说明这一行**还没做完 brainstorm**。两种选择并列，等于没选。

### 排除的替代列：**这张表的灵魂**

这一列是最容易被省略、也是威力最大的一列。

它的作用：

1. **防止 AI 在下一层把它们偷偷加回来**。你在 Vision 层排除了"移动 App 形态"，Design 层 AI 就不会问"要不要同时做移动端响应式"。
2. **把"为什么不选"固化下来**。三个月后你看到这一行，记得起当时排除的理由。
3. **提供可回滚的锚点**。未来真的需要用被排除的替代时，你知道当年是因为什么排除的，现在条件是否变化。

举个真实的例子。S1.1 Identity 的决策表里有一行：

```
| 限流 | 两层 (IP + 邮箱),仅计失败 | 账户锁定策略（易被 DoS） |
```

这个"账户锁定策略（易被 DoS）"就是排除的替代列的完整形态——**排除什么 + 为什么排除**。

未来哪天合规要求说必须有账户锁定，你一看这行，立刻知道：
- 当时没做是因为 DoS 风险
- 现在要做，必须同时设计防 DoS 机制（比如 CAPTCHA、IP 白名单逃生）
- 不能简单加一个"连续 5 次失败锁 30 分钟"了事

**没有这一列，你只会看到"限流策略 = 两层限流"。你不会想起为什么当时没做账户锁定，未来再做时就可能踩同样的坑。**

## 四、怎么让 AI 帮你填这张表

提示词模板：

```
我们要 brainstorm <项目/子系统>。

请按"关键决策表"格式逐维度追问我。表格三列：
| 维度 | 决策 | 排除的替代 |

规则：
1. 每次只问一个维度。
2. 先列出这个维度下的所有合理选项（至少 3 个）。
3. 逐个追问我："你倾向哪个？为什么？"
4. 我选定后，把其他选项整理到"排除的替代"列，每个加一句"为什么排除"。
5. 一个维度结束，才进入下一个。

从这些维度开始（欢迎你补充）：
- <维度 1>
- <维度 2>
- ...
```

关键在三个细节：

**细节 1 · 强制 AI "先列选项再追问"**

如果不强制，AI 会直接给你推荐一个答案。那样你失去了思考过程，"排除的替代"列会变成事后补的虚构物。

**细节 2 · 强制 AI "一个维度结束才进入下一个"**

AI 喜欢一次性把所有维度问完。那样的结果是每个维度都浅尝辄止。**慢就是快**。

**细节 3 · 强制"排除的替代"带理由**

只记录"排除了什么"没用，必须记录"为什么排除"。理由列在最后的 `说明` 或 `备注` 子列里。

## 五、什么时候该打开这张表

每一层的 brainstorm 都用它，但**表的粒度不同**：

| 层级 | 表的粒度 | 行数 |
|---|---|---|
| L1 Vision | 平台级决策（定位、架构、分发模型） | 5-8 行 |
| L2 Design | 子系统级决策（存储、协议、算法） | 10-20 行 |
| L3 Plan | 通常不需要（Plan 层是执行）| 0-3 行 |
| L4 Implementation | 不需要（对着 Design 填空）| 0 |

**一个常见错误**：L1 的表里写 L2 的决策。

比如你在 L1 Vision 表里写"数据库用 MySQL"。错了。L1 表应该是"持久层策略：自托管关系型数据库"——至于具体是 MySQL 还是 Postgres，放到 L2 Design 再决定。

**为什么**：L1 的决策一旦定死，所有 L2 都受约束。"MySQL" 是 L1 级别的约束吗？不是。"自托管关系型数据库" 才是——因为它排除了"云数据库"、"NoSQL"、"用户自带数据库"等 L2 候选。

## 六、表之外：两个增强写法

### 增强 1 · 加一列"演进锚点"

复杂项目里可以在决策表加第四列，叫"演进锚点"：

```
| 维度 | 决策 | 排除的替代 | 演进锚点 |
```

演进锚点回答的是：**什么条件变了，这一行需要重评？**

例如：
```
| 限流存储 | 内存 LRU | Redis、MySQL | 部署多实例时重评 |
```

三个月后你部署多实例，一眼看到这行要改。不用全表扫一遍。

### 增强 2 · "排除的替代"里可以带"什么时候会重新考虑"

```
| 会话机制 | Opaque token + sessions 表 | JWT (跨服务调用需求出现时重评) |
```

这样"排除"不是永远的，而是**"当前排除，未来在特定条件下可以重新讨论"**。把条件写清楚，未来不用靠记忆。

## 七、把这个方法迁移到非开发场景

这张表不只适用于技术决策。

**产品定位场景**：

```
| 维度 | 决策 | 排除的替代 |
| 用户类型 | C 端独立消费者 | B 端企业客户、创作者群体 |
| 收费模式 | 订阅制 | 一次性买断、免费增值、广告 |
| 获客渠道 | 内容营销 | 付费投流、KOL 合作、SEO |
```

**团队决策场景**：

```
| 维度 | 决策 | 排除的替代 |
| 远程协作工具 | 飞书 | 钉钉、Teams、Slack |
| 代码审查策略 | 必须 PR + 一人 approve | 直接 push、不要求 approve |
```

只要有"多个选项 → 选一个 → 会被质疑"的场景，都可以用。

## 八、总结

一张三列表：维度 / 决策 / 排除的替代。

- **维度列**：抽象的问题，不随技术栈变
- **决策列**：斩钉截铁，不含"也"和"可能"
- **排除的替代列**：这张表的灵魂——排除什么 + 为什么排除

使用的两个铁律：

1. 让 AI **一个维度一个维度问**，不要一次性问完
2. 每个排除项必须**带理由**，否则等于没写

三个月后当你回头看这张表，或者当 AI 在下一层想把排除项偷偷加回来时，这张表就是你最后的防线。

下一篇我会讲一个姐妹话题：**"明确不做什么"——为什么非目标比目标更有信息量**。
