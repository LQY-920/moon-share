## 图 2 · 模板目录

**标题**：
这个 12 节目录是谁的？

来自真实项目 S1.1（账户与身份）子系统的 Design 文档：

```
一、定位与边界
二、关键决策
三、项目结构
四、数据模型
五、API 契约
六、会话机制
七、安全与审计
八、错误处理
九、测试策略
十、依赖清单
十一、演进路径
十二、未决项
```

每节下面会讲它在防什么、少了会出什么问题。

---

## 图 3 · 第一节：定位与边界

**标题**：
① 定位与边界

**防**：AI 把邻居子系统的功能夹带进来

标准结构：
✅ M0 现实目标（这次做到哪一步）
✅ "不在本模块做的事"表（每项带归属）
✅ 前置/后置依赖

```
| 事项                         | 归属              |
|-----------------------------|------------------|
| 审计事件的订阅与存储         | S1.3 Audit       |
| 前端登录/设置页面            | 前端形态（L4）    |
```

少了这节 → 三周后你忘了边界在哪，两边各做一半

---

## 图 4 · 第二节 + 第三节

**标题**：
② 关键决策 | ③ 项目结构

**② 防**：Design 变成技术细节清单，不写设计决策记录

→ 三列表：维度 | 决策 | 排除的替代

**③ 防**：设计与实现脱节，目录结构各写各的

→ 文件树 + 依赖方向约束（只下行不反向）+ 对外暴露清单

---

## 图 5 · 第四节：数据模型

**标题**：
④ 数据模型

**防**：DDL 只写字段，不写设计意图

标准结构：
✅ DDL（直接贴）
✅ 每字段/约束 → "为什么"（这是 AI 最不会主动写的）
✅ 迁移策略

```
- sessions.token_hash 只存哈希：
  泄库时 token 不可逆推
- sessions.revoked_at 软删：
  保留审计轨迹，30 天后清理任务物理删除
```

少了"为什么" → 有人把 password_hash 加 NOT NULL，破坏 OAuth 账户

---

## 图 6 · 第五节：API 契约

**标题**：
⑤ API 契约

**防**：API 设计时没有统一错误码约定

标准结构：
✅ 统一错误响应格式
✅ 端点清单（含 M0 状态）
✅ 错误码字典

```
INVALID_CREDENTIALS  401  登录失败的唯一外显原因
NOT_FOUND            404  不泄露"用户是否存在"
```

少了错误码字典 → 各端点各自为政，客户端难处理

---

## 图 7 · 第六节 + 第七节

**标题**：
⑥ 会话机制 | ⑦ 安全与审计

**⑥ 防**：会话逻辑散落各处，状态转换没有统一记录

→ 每个关键流程画状态转换（登录/登出/过期/被踢）

**⑦ 防**：安全策略没有整体视图，EventEmitter 定义没记录

→ 安全设计整体视图 + AuthEvent 定义 + 发布事件清单

少了 ⑦ → S1.3 审计接入时找不到接口

---

## 图 8 · 第八节 + 第九节

**标题**：
⑧ 错误处理 | ⑨ 测试策略

**⑧ 防**：三层策略（domain/service/express）没有统一约定

→ 明确每层职责 + 防信息泄露约定（500 不透 SQL/栈）

**⑨ 防**：测试没有分层，安全路径漏测

→ 三层测试表 + M0 通过条件 + 明确不做的测试

```
M0 必须有：
✅ 密码错误不区分"邮箱不对/密码错"
✅ 限流返 429 带 Retry-After
✅ 改密码踢光所有会话
✅ 踢别人会话返 404
```

---

## 图 9 · 第十节 + 第十一节

**标题**：
⑩ 依赖清单 | ⑪ 演进路径

**⑩ 防**：Plan 阶段重复做"用什么"决策

→ 运行时依赖 + 开发依赖完整清单

**⑪ 防**：子系统被设计成"死结构"，一扩展就推倒

→ "未来动作 | 需要修改 | 不需要修改"表

```
| 接入 GitHub OAuth    | 新增 providers/github.ts | users / sessions 不动 |
| 多实例部署           | 限流换 Redis              | API 契约、表结构不动   |
```

**⑪ 是 Design 层最重要的节**。写不出来说明数据模型设计还有问题。

---

## 图 10 · 第十二节 + 自检表

**标题**：
⑫ 未决项 + 自检表

**⑫ 防**：文档假装"所有都定死了"，其实有些地方没想清楚

```
| 未决项 | 当前状态 | 决定方式 |
| 单仓 vs monorepo | 未定 | 先按 monorepo 布 |
```

---

**设计文档自检清单（收藏版）**：

□ 定位与边界：有 M0 目标 + "不做"表 + 依赖
□ 关键决策：三列表，每排除项带理由
□ 项目结构：文件树 + 依赖方向约束 + 对外暴露
□ 数据模型：DDL + 每字段"为什么" + 迁移策略
□ API 契约：统一错误格式 + 端点清单 + 错误码字典
□ 会话机制：每个关键流程的状态转换
□ 安全与审计：整体视图 + EventEmitter 定义
□ 错误处理：三层策略 + 防信息泄露
□ 测试策略：三层 + M0 通过条件 + 明确不做
□ 依赖清单：运行时 + 开发依赖完整
□ 演进路径：未来动作表，每项"需要修改"列够短
□ 未决项：有清单 + 决定方式

标题：
一份 Design 文档的 12 节模板，每个节防一个坑 📋

正文：
设计子系统时，最怕文档没结构——想到什么写什么，三个月后回来完全看不懂。

最近把一份真实项目的 Design 文档目录拆解了一遍，发现它可以作为模板复用 👇

12 节，每节都对应一个真实的坑：
① 定位与边界 → 防 AI 夹带邻居功能
② 关键决策 → 防 Design 变成技术清单
④ 数据模型 → 防 DDL 不写"为什么"
⑤ API 契约 → 防错误码不统一
⑪ 演进路径 → **这是 Design 层最重要的节**

10 张图逐节讲清，最后一张是自检表，可以收藏对照 ✅
