# 超级个体的工作流：4 个 Claude Code 会话同时在跑

> 本文是"moon-share"系列第 06 篇。前五篇都在讲"怎么想清楚"，这一篇讲"怎么动起来"。
> 配套图：一张电脑截图（见文章顶部），四个 Claude Code 窗口同时在跑。

## 一、先看这张图

> （此处配图：四窗口终端截图，四个红色标签分别是：内容分享/知识沉淀 · 仓库 · dev · plan）

这是我电脑晚上 10 点的真实截图。

四个终端窗口，四个 Claude Code 会话同时在跑。**我本人没有在写代码、没有在画图、没有在写文章**。我在做的只有一件事：看它们跑，在关键节点做决定。

这就是我理解的**超级个体** —— 不是"一个人能做很多事"，而是"一个人 + 几个 AI 并行地做很多事"。

本文把这张图拆开讲：**四个会话各自在做什么、它们怎么协同、我作为人在这个系统里扮演什么角色**。

## 二、四个会话的分工

按图上红色标签，从左上顺时针：

### 左上 · 内容分享/知识沉淀（对外）

这个会话在做的事：**把本周的开发经验沉淀成对外内容**。

具体来说，它正在：
- 读过去两周生成的 spec 文档（Vision、Design、Plan）
- 从中抽取可复用的方法论
- 产出公众号/小红书/视频号三种渠道的稿件

这是**唯一一个对外方向**的会话。其他三个都是对内（开发产品本身）。

**为什么这个会话必须独立开**：
- 它读的上下文和写代码完全不一样（需要的是叙事能力，不是架构能力）
- 它服务的读者不是编译器，是人类
- 它不需要看代码仓库的 git log，但需要看设计文档

### 右上 · 仓库管理（对内 · 把关）

这个会话在做的事：**review spec 文档、提交仓库**。

它刚刚完成了一件事：
- review 了一份刚写完的子系统 Design 文档
- 确认内容质量
- 等待我决定是否进入 writing-plans 阶段

这个会话承担的是**"把关人"**的角色。它不写代码、不写文档，它只负责**验收**。

**为什么把 review 独立成一个会话**：
- 如果让写代码的会话自己 review 自己，它会 biased（自己写的自己觉得好）
- 独立会话有新鲜的上下文，能看出盲点
- 让审查和创作在不同的"心智模式"下进行

### 左下 · dev（对内 · 生产）

这个会话在做的事：**subagent 自动跑 21 个 Task 写代码**。

这是**纯生产线**。我已经把 plan.md 交给它，它正在按照 Subagent-Driven Development 流程，一个 Task 接一个 Task 地写：
- Task 2：`core/config.ts`
- Task 3：`core/logger.ts`
- Task 5：`core/db.ts`
- Task 7：`domain/` 层
- Task 10：`password.service`
- ……

每个 Task 都有验收标准。写完一个 → 审查一个 → 进入下一个。**完全自动，我只看进度**。

这是整个系统里最"工业化"的会话。它的特点：
- 有明确的 plan.md 作为工作指令
- 每个 Task 有明确的输入输出
- 失败时会自动回退到上一个已验证的状态

### 右下 · plan（对内 · 规划）

这个会话在做的事：**规划下一步该做什么**。

它正在思考的问题：
- S1.1 Identity 目前做到 Task 15/21，预计还需要多久？
- 做完 S1.1 之后，下一步是 S2.1 记忆存储，还是先回头 review 现有代码？
- 另一条会话线（moon-share 内容产出）做完后怎么归纳到 moon-skills？
- 要不要开一个新的 git worktree，让 S2.1 可以提前并行启动？

这是**指挥所**的会话。它不生产、不审查，它负责**编排**。

## 三、这套分工背后的原则

看起来只是"开四个窗口"，但**为什么要这样分**、而不是"开一个窗口做所有事"？

### 原则 1 · 按"心智模式"切分，不按"功能"切分

每个会话对应一种**心智模式**：

| 会话 | 心智模式 | 核心能力 |
|---|---|---|
| 内容分享 | 讲述者 | 把复杂的事讲清楚给人类 |
| 仓库管理 | 审查者 | 找出盲点和质量问题 |
| dev | 工匠 | 按规范把代码写对 |
| plan | 指挥官 | 统筹全局、排优先级 |

这四种心智模式**不能混在一个会话里**。混在一起，AI 会在"创作态"和"审查态"之间来回切换，上下文被污染。

### 原则 2 · 上下文不重叠

四个会话加载的上下文不同：

- **内容分享**：读 spec 文档 + 以前发布的文章（找风格）
- **仓库管理**：读新产出的 spec + git diff
- **dev**：读 plan.md + 当前代码库
- **plan**：读所有上述内容的摘要（不读细节）

如果让一个会话承担所有事，它的上下文窗口会爆炸，关键信息被稀释。

### 原则 3 · 人是编排者，不是执行者

**我在这个系统里扮演的角色**：

- ❌ 我不写代码
- ❌ 我不写文章
- ❌ 我不 review 每一行输出
- ✅ 我在**关键决策点**出现：选题方向、架构拍板、是否放行

四个会话加起来一天能产出的东西，比我一个人写一周还多。我的瓶颈不在"手速"，在"决策速度"。

## 四、它们怎么协同？

四个会话不是独立的，它们之间有数据流转：

```
       ┌──────────────────────────────┐
       │  plan（规划下一步）           │
       └─────────┬────────────┬───────┘
                 │            │
                 ▼            ▼
       ┌─────────────┐  ┌──────────────┐
       │ dev（生产） │  │ 内容分享（对外）│
       └──────┬──────┘  └──────────────┘
              │
              ▼
       ┌──────────────┐
       │ 仓库管理      │
       │（review & 提交）│
       └──────────────┘
```

**规则 1 · plan 产出 plan.md，dev 按 plan.md 执行**

plan 会话不直接命令 dev 会话做事。plan 的产出是一份 plan.md（或者简短的执行路径），dev 会话读这份文件后自己开工。

**这种"文档作为接口"的方式**非常关键——它意味着 dev 会话可以在任何时候停下来、保存、下次继续，不需要"记住 plan 当时说了什么"。

**规则 2 · dev 完成阶段性任务后，仓库会话 review**

dev 不直接提交代码。它把代码写好 → 通知我 → 我让仓库会话去 review → review 通过 → 仓库会话提交 git。

这是两阶段审查：dev 自己测试是第一层，仓库会话跨会话 review 是第二层。

**规则 3 · 内容分享会话独立跑**

内容分享会话**不介入开发流程**。它只读已经定稿的 spec 文档，产出对外文章。

这个隔离很重要。对外内容的节奏比对内开发慢得多（今天写的文章可能下周才发），如果让它卡在开发流程里，两边都会被拖累。

**规则 4 · plan 会话定期同步所有会话的状态**

每天或每个阶段，plan 会话会"巡检"：
- dev 进度到哪了？
- 仓库里有没有 pending 的 review？
- 内容分享写到第几篇了？

然后决定**下一个阶段的重心**。

## 五、实操：怎么开始这套工作流？

不需要一开始就开四个窗口。**渐进式推进**：

### 阶段 1 · 单会话（最常见）

一个 Claude Code 会话做所有事。你和 AI 一来一回，AI 帮你写代码、审查代码、规划下一步。

**适合**：小项目、探索阶段、一次性脚本。

### 阶段 2 · 双会话：dev + plan

开两个窗口：
- 一个做 plan（写文档、规划）
- 一个做 dev（按文档写代码）

**关键动作**：plan 产出的文档必须**写到磁盘**（不是只在对话里），dev 会话读这个文档开工。

**适合**：中等规模项目，有多个独立子系统，需要提前规划。

### 阶段 3 · 三会话：加上 review

再开一个窗口做 review。dev 每完成一个阶段，让 review 会话**从零开始读代码**（不加载开发时的对话上下文）。

**关键动作**：review 会话不能是 dev 会话的延续，必须是**新开的窗口**，才有"新鲜视角"。

**适合**：需要代码质量保证的项目、将来要给别人用的代码。

### 阶段 4 · 四会话：加上对外内容

当项目本身产出了足够多的经验、值得沉淀成对外内容时，开第四个会话。

**关键动作**：这个会话读的是 spec 文档和已发布的文章，**不读代码**。它的职责是"把复杂的事讲清楚"，不是"把代码讲清楚"。

**适合**：有持续输出意愿、想打造个人品牌的开发者。

## 六、常见误区

### 误区 1 · "开四个窗口 = 更快"

不一定。如果你的决策速度跟不上，四个窗口会都在等你，反而更慢。

**真正的提速来自"并行"**：一个窗口在跑 30 分钟的自动化任务时，你可以和另一个窗口讨论下一步。手动干预的时间不变，但空闲时间被利用了。

### 误区 2 · "让 AI 自己管所有窗口"

可行，但代价高：
- AI 之间的通信成本高（每个窗口的输出都要被另一个窗口读懂）
- 没有人类决策点，出错时损失更大

**更好的方式**：人做编排者（四个窗口的指挥），AI 做执行者（每个窗口里的具体工作）。

### 误区 3 · "必须实时同步"

不需要。各会话之间**通过磁盘文件同步状态**：plan 写 plan.md，dev 读 plan.md；dev 写代码，review 读代码。

这种异步协同的好处：**任何一个会话崩溃/失联，都不影响其他会话**。

## 七、这对"超级个体"意味着什么

**个体**这个词，以前默认是"单线作战"。

一个人只能有一双手、一双眼、一个大脑。你同时只能做一件事。

AI 改变的不是"你的手速"——你写代码快一倍、写文章快一倍，但仍然是**单线**。

AI 真正改变的是**你能并行的工作数**。

一个人 + 4 个 AI 会话 = 一个小团队的产能：
- 一个开发
- 一个测试/审查
- 一个产品经理
- 一个内容/市场

以前这需要 4 个人、一套协作工具、无数会议。

现在这需要一台电脑、4 个 Claude Code 会话、一套清楚的分工规则。

**这才是超级个体**。不是"个人英雄主义"，是"**个人 + AI 团队**"。
