# 从一篇行业文章到 28 个子系统：一次完整的平台愿景产出过程

> 本文是"moon-share"系列第 05 篇。
> 姊妹篇是 [01 四层设计法](../01-四层设计法/gzh.md)，讲的是四层抽象的方法框架。本文是它的实证——看一个具体项目怎么从零走到 Vision 文档。

## 一、本文的故事线

我读了一篇钉钉陈航的演讲文章（关于"日抛型软件"），觉得这个概念有意思。

然后我用 AI 做了一次 brainstorm，一个多小时，产出了一份完整的**平台愿景 + 子系统拆解文档**——5 层、28 个子系统、DAG 依赖图、6 个里程碑实现顺序。

这篇不是方法论文章，是**叙事文章**。我把这个完整过程讲一遍：我是怎么想的、怎么问的、AI 给了我什么、我又改了什么。

中间穿插方法论的解释，但不是为了讲方法论，是为了讲清楚为什么在那个节点我要那样做。

**适合谁读**：已经看过 [01 四层设计法](../01-四层设计法/gzh.md)，想看看"一个真实项目怎么从行业文章走到 Vision 文档"的人。

## 二、起点：一篇让我停下来的文章

那是 4 月初，我在刷行业资讯，看到一篇钉钉陈航的演讲整理，主题是"日抛型软件"。

核心观点就一句话：
> 软件按需生成、用过即毁，但使用它所产生的知识和记忆持久留存并持续进化。

这句话击中了我的原因：

- 我一直在想"AI 能不能帮我在几分钟内搭一个工具，用完就扔"
- 但"用完就扔"的问题是：每次都得重新教 AI 上下文
- 如果记忆能跨"日抛"持续进化，那每次生成的质量会按日提升

**我的反应不是"这个概念很酷"，而是"这个概念能落地成一个产品"**。

但当时我没有把它变成 Vision 文档。我只是记了笔记，然后继续做其他事。

## 三、两周后：我想认真做这件事了

两周后的一个晚上，我决定把"日抛型软件"这件事做成一个真实项目。

但我没有直接让 AI 写代码。我先花了半小时做一件事：

**问自己：这到底是一个什么项目？**

这个问题看起来很蠢——"就是日抛型软件平台啊"。

不对。这个问题下面还有一层：**用户是谁？他们为什么用这个而不是其他方案？**

我试着回答：

- 用户是：需要按需工具的人，但不希望每次从零开始
- 核心区别：如果我用完一个工具就删掉了，下次重新生成时，AI 记得我之前用过什么、为什么用过、有什么坑——这才是差异化

这才是平台的**核心理念**。

不是"日抛软件"，而是**"软件日抛、记忆进化"**。

这一句话花了我 20 分钟。但它是整个 Vision 文档的起点。后面所有东西都从这里长出来。

## 四、第一次 AI 对话：让 AI 帮我拆解

有了核心理念，我决定开始第一次 brainstorm。

不是让 AI 直接设计系统，而是让 AI 帮我**追问**：还有哪些维度没想清楚？

我给 AI 的提示词大概是这样：

```
我有一个想法：一个"日抛型软件 + 记忆进化"的 AI 平台。
核心理念：软件日抛、记忆进化。
目标用户：需要按需工具的人。
请帮我逐维度追问：
1. 这个平台的"日抛"具体指什么？（软件形态按需生成、用完即毁？还是按日进化？）
2. "记忆"存储在哪里？谁的记忆？
3. 平台提供哪些形态的工具？
4. 这个平台和 Claude Code 这样的工具怎么区分？
```

AI 问了我很多问题。有些我想清楚了，有些没有。

**没有想清楚的，我就先记为"未决项"，不在这个阶段硬做决定**。

比如"平台的分发形态"——一开始我想的是"单个用户自己用"，但 AI 问"如果有人想把自己做的工具分享给别人呢"，我想了一下，说"长期应该有市场，但初期不重要，先记为远期目标"。

这个决策后来进了 Vision 文档的"关键决策表"：

```
| 分发模型 | 有公共市场（类 GPTs Store） | 创作者生态作为长期愿景的一部分 |
```

不是这个阶段拍板了"必须有市场"，而是**"长期有，短期先不做，但结构上要预留"**。

## 五、关键决策表：把拍板的决定固化下来

第一次对话结束后，我整理了一份"关键决策表"。

当时写的第一版：

```
| 目标用户    | 通用产品/平台（给别人用）                   |
| 日抛语义    | 按日进化                                   |
| 技术底座    | 构建在现有生态之上（编排层）                |
| 多形态关系  | 意图优先 + 形态优先 双路径                  |
| 记忆模型    | 用户级统一记忆中枢                         |
| 分发模型    | 有公共市场（类 GPTs Store）                 |
```

这张表花了我 15 分钟。每一行都是一个"我拍板了、不再讨论"的决定。

**为什么要写这张表，而不是直接在脑子里想？**

因为脑子里的决定是模糊的、会被时间侵蚀的。写下来才能检验：
- 两个决策之间有没有矛盾？
- 某个决策会不会让另一个决策变得不可能？

比如"技术底座：构建在现有生态之上"和"目标用户：通用产品/平台"——这两个决策加在一起，意味着我**不做自己的大模型、不做自己的 MCP 协议**，这些全部依赖外部生态。

这个约束一写出来，我立刻知道：接下来拆子系统时，"模型调度"不在我的职责范围内，"MCP 集成"才是。

## 六、顶层架构：从一个问题开始

核心问题：**平台分成几层、每层负责什么？**

AI 给我的第一个版本是"前端 / 后端 / 数据库"，太传统了。

我说不对。这个平台的独特性在于"记忆"。记忆在中间层，还是底层？

我思考了一下：

- 如果记忆在底层，那上面所有东西都依赖它，但"日抛"意味着上面可以随时换
- 如果记忆在中间，上面是产物层（可以日抛），下面是治理层（账户/权限，不常换）

我画了第一个草图：

```
上层：产物层（日抛）
中层：记忆层（进化）
底层：治理层（账户/权限/审计）
```

AI 帮我把这个画成了"四层洋葱"：

```
         ┌──────────────────────────────────────┐
         │     L5 分发层 Marketplace & Share      │
         ├──────────────────────────────────────┤
         │     L4 形态层 Surfaces                │
         │     （app/网页/MCP/Skill/Agent...）    │
         ├──────────────────────────────────────┤
         │     L3 生成 + 运行引擎 Forge & Runtime│
         │     （意图→产物、调度、迭代循环）       │
         ├──────────────────────────────────────┤
         │     L2 记忆中枢 Memory Core           │
         ├──────────────────────────────────────┤
         │     L1 治理底座 Governance            │
         │     （账户、权限、审计、计费）          │
         └──────────────────────────────────────┘
```

这个图花了我和 AI 来回讨论大约 40 分钟。

关键的调整是：AI 第一次画的版本没有 L5（分发层）。我加上去的，因为我知道长期要有市场，但 AI 不知道。

**AI 的视觉是局部的，你的视觉是全局的。架构图必须你来拍板，AI 只能帮你画、帮你提问题。**

## 七、拆子系统：从"能单独立项"开始

四层画出来了。接下来要把每层拆成具体的子系统。

原则只有一条：**每个子系统都能单独立项，失败不拖垮邻居**。

AI 给的第一次版本是"身份管理、权限管理、记忆存储……"，太粗了。比如"权限管理"是一件事还是两件事？"权限策略引擎"和"权限评估"是不是应该分开？

我花了大约 30 分钟逐层过了一遍，最终产出 28 个子系统：

- L1（治理底座）：4 个
- L2（记忆中枢）：5 个
- L3（生成+运行引擎）：6 个
- L4（形态层）：8 个
- L5（分发层）：5 个

这里有一个判断：**为什么 L4（形态层）有 8 个子系统，而 L5（分发层）只有 5 个？**

因为 L4 的每种形态（网页/App/小程序/MCP/Skill/工作流/Agent/形态网关）都是独立的技术路径，彼此之间不共享实现。"网页"和"App"没有多少可以复用的代码，所以各算各的子系统。

但 L5 的"市场目录"和"发布流水线"共享同一个后端基础设施，所以合并成少数几个子系统。

**子系统的粒度不是均匀的。粒度取决于"这块东西失败时会影响谁"。**

## 八、DAG 依赖图：画完子系统之后最重要的事

28 个子系统列出来了，但它们不是并列的——有依赖关系。

我让 AI 帮我画 DAG（Directed Acyclic Graph，有向无环图）。AI 画的第一次版本把几乎所有东西都指向了 L1（身份）。

我说是的。但有些东西也指向 L2（记忆）。比如 Forge（生成引擎）既依赖身份，又依赖记忆——因为生成时要读记忆（上下文），生成后要写记忆（反馈）。

**这张图最重要的发现是：L3（Forge）是整个平台的核心枢纽**

它依赖 L1（身份）和 L2（记忆），又被 L4（形态层）和 L5（分发层）依赖。

如果 Forge 做得烂，整个平台塌一半。所以实现顺序上，Forge 要先做，而且要做稳。

## 九、实现顺序：从"能停下来验证"出发

有了 DAG，我可以排实现顺序了。

我的原则是：**每个里程碑都是能停下来验证的自洽切片**。

如果第一个里程碑做完了，系统不能跑，那第二个里程碑就没有基础。

按这个原则，第一个里程碑（M0）只能是：L1 治理底座的最小子集（S1.1 身份 + S1.2 策略）。

为什么？因为后面所有东西都要挂在用户身上。

M0 做完，你有一个可以注册/登录的系统，有"我的数据属于我"的基础隔离。

M1 是 L2 记忆骨架。为什么在 L4（形态）之前做记忆？因为没有记忆，所有产物都是一次性的，不符合"按日进化"定位。

M2 是 L3 骨干 + L4 第一形态。为什么形态排在 L2 之后？因为形态消耗的是记忆，记忆没有的话形态就没有意义。

……

6 个里程碑排出来，第一个是"地基"，最后一个是"分发 + 治理补齐"。

## 十、战略性说明：那些不在这次实现范围里的东西

写完以上所有内容，最后一节我专门写了"暂不纳入的讨论点"：

```
暂不纳入的讨论点（保留但不立项）：
- 独立的开发者/扩展 SDK（给平台写插件）— 近期不独立，按需嵌入各子系统
- 独立的通知/触发器子系统 — 归入 S4.6 工作流 和 S4.7 Agent
- 独立的多端同步/离线子系统 — 归入 S2.1 存储 和 各 L4 形态
```

每一项都写清楚了**为什么排除**和**归谁承接**。

**这一节的价值**：把"不是现在做的"从脑子里移出来，写到纸上，然后打上"保留但未立项"的标签。

这样，AI 在 Design 层、Plan 层、Implementation 层都不会把"SDK 插件"当作候选功能推给我——因为白纸黑字写了"暂不纳入"。

## 十一、写完 Vision 文档之后发生了什么

写完这份 Vision 文档的第二天，我开始做第二个子系统（S1.1 账户与身份）的 Design 文档。

Design 文档写起来比 Vision 快得多——因为 Vision 回答了"我要做什么平台"，Design 只回答"这个子系统长什么样"。

Design 文档写了 3 个小时（包含和 AI 的来回讨论），产出了 12 节、6000 多字的文档。

如果没有 Vision 文档，Design 文档的第一节"定位与边界"就得从零想：S1.1 到底做什么、不做什么？

有了 Vision 文档，这节的答案几乎是现成的——"S1.1 是 L1 治理底座的身份子系统，负责注册/登录/多设备会话，预留 OAuth 接口。"

**Vision 是 Design 的前提。不是哲学，是效率。**

## 十二、这次 brainstorm 的几个可复用经验

### 经验 1：从"核心一句话"开始，不从"系统列表"开始

很多人在开始一个新项目时，会先列功能清单："要有登录、要有记忆、要有网页生成……"

这不是 Vision。这是 to-do list。

Vision 的起点是一个**一句话能讲清楚的核心定位**。然后这个核心定位向下长出架构、向上约束形态。

"软件日抛、记忆进化"——这句话决定了：
- 形态必须是可日抛的（网页/App/MCP 都可以，定制开发的 native App 不行）
- 记忆必须是跨形态共享的（不能是每个形态各自存各自的记忆）

没有这句话，你的设计决策就没有锚点。

### 经验 2：关键决策表要在 Vision 层固化，Design 层细化

Vision 层的决策表粒度是"平台级约束"（比如"技术底座 = 编排层"），Design 层的决策表粒度是"子系统级技术选择"（比如"会话机制 = Opaque token"）。

两者不能混。Vision 层如果写了"用 Node.js"，Design 层的所有子系统都被迫用 Node.js——这是过度约束。

### 经验 3：子系统粒度取决于"失败时影响谁"

28 个子系统不是均匀分布的。L4（形态层）有 8 个，因为每种形态都是一个独立的技术路径。L1（治理底座）只有 4 个，因为身份/权限/审计/计费共享同一个基础设施。

**判断标准：如果这块东西失败了，会不会拖垮其他子系统？**

会 → 它应该是自己的子系统。
不会 → 它应该归入邻居子系统。

### 经验 4：里程碑要能停下来验证

6 个里程碑，每个完成后都有可演示的产出：

- M0 完成后：可登录的系统
- M1 完成后：可存取的记忆（虽然只是文本仓）
- M2 完成后：第一个可运行的产物（从意图到网页）
- M3 完成后：同一意图第二次比第一次明显更好

如果某个里程碑完成后没有可演示的产出，说明这个里程碑的粒度太大，需要拆。

### 经验 5：战略说明节是防止 AI 乱塞功能的最重要工具

这是很多人在 brainstorm 时忽略的。

AI 在 Design 层、Plan 层、Implementation 层都会倾向于把"看起来合理"的功能塞进来——因为它不知道你排除了什么。

把"暂不纳入的讨论点"写进 Vision 文档，是给 AI 的全局约束。以后 AI 每次推一个功能进来，你先检查 Vision 文档——如果 Vision 里写了"暂不纳入"，直接拒绝。

## 十三、尾声

这次 brainstorm 从晚上 8 点到 11 点，总共大约 3 小时。

产出：
- 一份 3000 字的 Vision 文档（5 层、28 个子系统、DAG、6 个里程碑）
- 一份 6000 字的 Design 文档（12 节）
- 一份 1000 字的 Plan 文档（第一步已执行）

如果跳过 Vision 直接写 Design，我估计 Design 文档会多花 2-3 倍的时间，而且设计质量会更差。

因为没有 Vision，Design 的第一节"定位与边界"就不知道该从哪里划。
