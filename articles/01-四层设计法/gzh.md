# 四层设计法：AI 时代做一个平台级项目，该怎么分层推进

> 本文是"moon-share"系列的第 01 篇，也是后续所有方法论文章的锚点。
> 配套材料：[moon-agent-os 平台愿景文档](https://github.com/LQY-920/moon-agent-os/blob/main/docs/superpowers/specs/2026-04-22-moon-agent-os-vision-decomposition.md) 、 [S1.1 Identity 子系统设计文档](https://github.com/LQY-920/moon-agent-os/blob/main/docs/superpowers/specs/2026-04-23-s1-1-identity-design.md)

## 一、先讲一个常见翻车现场

你有一个想法——做一个 AI 产品、一个平台、甚至只是一个稍微复杂点的内部工具。

你打开 Claude Code 或者 Cursor，直接让它开干：

> "帮我搭一个用户登录系统，用 Node.js + MySQL，带邮箱密码登录和多设备会话。"

AI 立刻给你生成代码。跑起来了，你开心。

三天后，你想加"微信扫码登录"。你发现需要把现在的 session 机制推倒重写，因为之前生成的代码把 `session` 和 `password` 耦合死了。

一周后，你想把这个登录系统搬到另一个项目复用。你发现当时的代码假设了"只有一个应用"，没有组织、没有租户概念，又要重写一遍。

一个月后，你开始怀疑：**是不是我一开始就不该让 AI 直接写代码？**

是。

但真正的问题不是"AI 写得不好"。是**你在错误的抽象层次上让 AI 开工**。

## 二、AI 时代，一个平台级项目的四层抽象

我最近在做一个叫 moon-agent-os 的平台（细节不重要），在推进的过程中形成了一套我自己用得顺手的分层法：

```
          ┌──────────────────────────┐
          │  L4 · Implementation     │  写代码、跑测试、debug
          ├──────────────────────────┤
          │  L3 · Plan               │  具体任务分解，按小时/天排期
          ├──────────────────────────┤
          │  L2 · Design             │  单个子系统的数据模型/API/边界
          ├──────────────────────────┤
          │  L1 · Vision             │  平台定位、架构、子系统清单
          └──────────────────────────┘
```

每一层解决不同的问题，产出不同形态的文档，用不同的提问方式和 AI 协作。

跳过上层直接做下层，结果就是你开头看到的那个翻车现场。

下面我按 L1 → L4 逐层讲清楚。

## 三、L1 · Vision：平台在回答"我到底要做什么"

Vision 层的产出只有一份文档：**平台愿景 + 子系统拆解**。

它回答这些问题：

- 这个平台的核心定位是什么？（一句话能讲清）
- 关键约束是什么？（哪些维度必须定死，哪些可以浮动）
- 顶层架构长什么样？（几层、怎么切）
- 独立子系统有哪些？（每一个都能单独立项）
- 子系统之间什么依赖关系？（DAG）
- 实现顺序怎么排？（里程碑）

以我这个 moon-agent-os 的 Vision 文档为例，它的目录是这样的：

```
一、核心理念
二、关键决策（约束）
三、顶层架构：四层洋葱
四、独立子系统清单
五、依赖关系（DAG）
六、实现顺序（6 个里程碑）
七、战略性说明
八、后续工作指引
```

**这一层有两个容易被跳过但不能跳过的东西**：

### 1. 关键决策表

一张三列表：`维度 | 决策 | 排除的替代`。

```
| 维度       | 决策                      | 说明              |
|-----------|--------------------------|-------------------|
| 目标用户    | 通用产品/平台(给别人用)    | 必须包含账户、数据隔离、分发等完整平台能力 |
| 日抛语义    | 按日进化                  | 软件瞬时、记忆持久 |
| 技术底座    | 构建在现有生态之上(编排层) | 模型/MCP/Claude Code 等都作为底层依赖 |
| 多形态关系  | 意图优先 + 形态优先 双路径 | 两种用户入口并存 |
| 记忆模型    | 用户级统一记忆中枢         | 单一归属主体，跨产物共享 |
| 分发模型    | 有公共市场(类 GPTs Store) | 创作者生态作为长期愿景的一部分 |
```

这张表是 Vision 层最强的收敛工具。它不只说"你选了什么"，更重要的是**明示了你排除了什么**。关于这个表我会在下一篇单独写，这里先点一下。

### 2. 暂不纳入的讨论点

Vision 文档必须有一节专门写"**不做什么**"。

比如我这份 Vision 文档里有这么一段：

```
暂不纳入的讨论点（保留但不立项）：
- 独立的开发者/扩展 SDK —— 近期不独立，按需嵌入各子系统
- 独立的通知/触发器子系统 —— 归入 S4.6 工作流 和 S4.7 Agent
- 独立的多端同步/离线子系统 —— 归入 S2.1 存储 和 各 L4 形态
```

每一项都附了"为什么不立项"。这不是谦虚，是**防止 AI 在后续每一层都反复把这些东西推销给你**。Vision 层不明确，Design 层就会把它们夹带进去。

### L1 的输出长什么样

- 一份 Markdown 文档，通常 2000-5000 字
- 文件名格式：`YYYY-MM-DD-<platform>-vision-decomposition.md`
- 存放路径：`docs/superpowers/specs/` 或者项目的 `specs/` 目录
- **不包含任何代码、DDL、接口定义**——这些是下一层的事

### L1 阶段的 AI 协作方式

不让 AI 写代码。用 brainstorm 模式：

```
我要做一个 <一句话定位> 的平台。
核心理念是 <核心概念>。
帮我逐项追问：
1. 目标用户和边界
2. 关键的技术/产品决策
3. 顶层架构候选方案（至少 2 个，讲清楚选择理由）
4. 按照选定架构拆出独立子系统
5. 子系统依赖关系
6. 实现顺序（里程碑）
每一步都要问我"排除了什么"。
```

关键是：**让 AI 追问你，而不是让 AI 替你决定**。Vision 层的决定必须是你自己做的——AI 可以列选项，可以指出你的矛盾，但最后拍板的是你。

## 四、L2 · Design：单个子系统在回答"它具体长什么样"

当你的 Vision 文档里列出了一堆子系统（比如我这份里有 28 个），**选一个进入 Design 层**。

Design 层的产出是**每个子系统一份设计文档**。

它回答这些问题：

- 这个子系统的**定位**是什么？哪些事是它做的，哪些不是？
- 数据模型长什么样？为什么这么设计？
- 对外 API 是什么？错误码怎么定义？
- 它依赖谁、被谁依赖？
- 怎么测试？
- 未来扩展时哪些地方会动，哪些地方不会动？

以我这个 moon-agent-os 里 S1.1（账户与身份）的 Design 文档为例，目录是这样的：

```
一、定位与边界
二、关键决策
三、项目结构
四、数据模型
五、API 契约
六、会话机制
七、安全与审计
八、错误处理
九、测试策略
十、依赖清单
十一、演进路径
十二、未决项
```

比 Vision 层细得多，但**仍然不是代码**。

### L2 的几个关键节必须写

**"定位与边界"**：这一节划清这个子系统不做什么。比如 S1.1 的这节写了 7 条"不在 S1.1 做的事"：

```
| 事项                        | 归属                         |
|-----------------------------|------------------------------|
| 角色/权限策略引擎            | S1.2 Policy                  |
| 审计事件的订阅与存储         | S1.3 Audit (S1.1 只发布事件)  |
| 计费与配额采集               | S1.4 Billing                 |
| OAuth/SSO 具体 provider 接入 | 延后子系统 (S1.1 只留接口层)  |
| 邮箱发送服务                 | 延后（M0 只留占位路由）       |
| 前端登录/设置页面            | 前端形态（L4）负责            |
```

**没有这一节，AI 就会在实现阶段不断把邻居子系统的功能夹带进来**。

**"关键决策"**：又是那张三列表。比 Vision 层的决策表更细一级：

```
| 维度          | 决策                        | 排除的替代              |
|---------------|----------------------------|-------------------------|
| 登录基线      | 仅邮箱+密码                 | 手机号验证码、OAuth     |
| 会话机制      | Opaque token + sessions 表 | JWT、JWT + refresh      |
| 密码哈希      | argon2id                   | bcrypt、scrypt          |
| 限流          | 两层 (IP + 邮箱),仅计失败   | 账户锁定策略（易被 DoS） |
| ...           | ...                        | ...                     |
```

**"数据模型"**：直接贴 DDL。比如：

```sql
CREATE TABLE sessions (
  id            CHAR(26)      NOT NULL,
  user_id       CHAR(26)      NOT NULL,
  token_hash    CHAR(64)      NOT NULL,  -- sha256(原始 token) hex
  user_agent    VARCHAR(512)  NULL,
  ip            VARCHAR(64)   NULL,
  created_at    DATETIME(3)   NOT NULL,
  last_seen_at  DATETIME(3)   NOT NULL,
  expires_at    DATETIME(3)   NOT NULL,
  revoked_at    DATETIME(3)   NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uq_token_hash (token_hash),
  KEY idx_user_active (user_id, revoked_at),
  CONSTRAINT fk_sessions_user FOREIGN KEY (user_id) REFERENCES users(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

但**每个字段都要附上"为什么"**：

- `token_hash` 为什么存哈希不存原文？（泄库时 token 不可逆推）
- `revoked_at` 为什么用软删？（保留审计轨迹，30 天后清理任务物理删除）

AI 帮你写 DDL 很快，但**"为什么这么设计"的理由 AI 不会主动写**——你必须追问它，让理由白纸黑字写进文档。

**"演进路径"**：这一节列"未来要做什么动作时，哪些地方会改、哪些不会改"。比如：

```
| 未来动作               | 需要修改                      | 不需要修改          |
|------------------------|------------------------------|---------------------|
| 开放客户自助注册        | /auth/register 从 501 换实现 | users 表结构        |
| 接入 GitHub OAuth       | 新增 providers/github.ts     | users 表、session 机制 |
| 加组织/团队概念         | 新增 organizations 表        | users / sessions 不动 |
| 多实例部署              | 限流存储从内存换 Redis        | API 契约、表结构不动 |
```

**这一节是 Design 层最重要的节**。它强制你现在就想清楚"哪些是决定死的、哪些是可以演化的"。写不出来，说明你的数据模型设计还有问题。

### L2 的输出长什么样

- 每个子系统一份 Markdown 文档，通常 3000-8000 字
- 文件名格式：`YYYY-MM-DD-<sub-system-id>-design.md`
- 和对应的 Vision 文档放在同一目录
- **包含 DDL、API 契约、错误码、配置项，但不包含具体业务代码**

### L2 阶段的 AI 协作方式

```
读入 Vision 文档作为上下文。
我现在要设计 <子系统 ID>。
按以下目录逐节产出 design 文档：
一、定位与边界（必须列出"不做的事"）
二、关键决策（三列表）
...
每节完成前，明确追问我"你确定排除了 <X> 吗"、"<Y> 如果变了会影响这里吗"。
```

到了这一层，AI 开始深度参与。**但它仍然不写代码**。

## 五、L3 · Plan：把 Design 拆成可执行的任务

Design 写完了。它告诉你"这个子系统长什么样"，但没告诉你"一天做多少、按什么顺序做"。

L3 Plan 层回答的问题：

- 从 0 到 1 要走几步？
- 每步的**验收标准**是什么？
- 步骤之间什么依赖？
- 卡住了怎么办？

Plan 文档的基本结构：

```
步骤 1：初始化项目骨架
  - 行动项
  - 验收标准
  - 产出文件

步骤 2：数据库迁移
  ...

步骤 N：...
```

**每一步都要有验收标准**。"实现 session 机制" 不是验收标准，"能通过 9.1 节列出的 5 个安全路径测试用例" 才是。

Plan 层和 Design 层的分界线很清楚：

- Design 说"会话用 opaque token + sessions 表"
- Plan 说"第 3 步实现 SessionRepository，写完第 4 步 AuthService 调它"

Plan 文档通常不长（1000-3000 字），但**它是 Design 到 Implementation 的传送带**——没有它，AI 会从 Implementation 的第一行代码开始自由发挥。

## 六、L4 · Implementation：终于开始写代码

到这一层，AI 的发挥空间反而最小。因为：

- Vision 决定了它"做什么平台"
- Design 决定了它"写什么表、什么接口"
- Plan 决定了它"先写哪个文件、验收标准是什么"

它只需要填空：**按 Plan 的步骤、符合 Design 的契约、不偏离 Vision 的定位**。

这一层的 AI 协作方式最简单：

```
读入 <design.md> 和 <plan.md>。
现在执行 plan 的第 N 步。
写完后对照 Design 第 X 节的契约自检。
```

当 AI 在这一层"自由发挥"时（比如擅自给你加一个"找回密码"功能），你就知道问题出在哪——**上一层没有明确写"明确不做"**。

## 七、为什么不能跳层？

讲到这里你可能会想：小项目还用分四层吗？

不用。**一个只有一屏代码的脚本，你完全可以直接让 AI 写**。

但凡你的项目满足下面任何一条，就必须分层：

- 预计要迭代超过一个月
- 有第二个子系统或模块
- 需要提供对外 API
- 有数据库持久化
- 将来可能要给别人用

跳层的代价不是线性的，是**指数的**：

- 跳过 Vision → 永远在"加新功能就破坏老功能"的循环里
- 跳过 Design → 数据模型每加一个需求就要推倒
- 跳过 Plan → AI 每次写代码都从零开始设计，上下文不连续

AI 写代码的速度提升了 10 倍，但如果架构是错的，你要重写 10 次。净收益 0。

## 八、四层设计法自检表

这张表你可以直接存下来。下次开新项目前，从 L1 开始问自己：

```
L1 · Vision
[ ] 我能用一句话讲清这个平台的定位吗？
[ ] 我有一张关键决策表吗？（维度 | 决策 | 排除的替代）
[ ] 我列出了"暂不纳入的讨论点"和理由吗？
[ ] 我有顶层架构和子系统清单吗？
[ ] 我排了实现顺序（里程碑）吗？

L2 · Design（针对每一个要动工的子系统）
[ ] 我划清了"本子系统不做什么"吗？
[ ] 我的数据模型每个字段都附了"为什么"吗？
[ ] 我有 API 契约和错误码字典吗？
[ ] 我写了"演进路径表"吗？
[ ] 我列了"未决项"吗？

L3 · Plan
[ ] 每一步都有明确的验收标准吗？
[ ] 步骤之间的依赖清楚吗？

L4 · Implementation
[ ] 我让 AI 对着 Design 和 Plan 写，而不是让它自由发挥吗？
```

全部勾上再让 AI 开工。
